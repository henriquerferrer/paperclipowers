## 2. Decisions Locked (do not re-litigate)

These decisions were made on 2026-04-17 after investigating Paperclip's server source (heartbeat.ts, claude-local adapter execute.ts, Designer's Stage 6 ndjson log + context_snapshot). Each references the evidence that grounds it.

### 2.1 A10 path → shared project workspace (Path 3)

Paperclip's `resolveWorkspaceForRun` (heartbeat.ts:1407-1608) has four priority levels:
`project_primary` (via `projectWorkspaces` row) > `project_primary managed` (auto-created from projectId) > `task_session` (session-reuse) > `agent_home` (per-agent fallback at `/paperclip/instances/default/workspaces/<agent-id>`).

The adapter's `config.cwd` override **only applies when `source === "agent_home"`** (claude-local adapter execute.ts:149). With a Paperclip Project in play, `source === "project_primary"` and `config.cwd` is silently ignored. In the Stage 6 test company, all issues had `project_id = NULL`, so every run resolved to `agent_home` — each agent got its own UUID-keyed directory, with no cross-agent code sharing. Designer found Engineer's work by filesystem-snooping (`ls /paperclip/instances/default/workspaces/`), which is not a designed mechanism.

Stage 6.5 adopts the `project_primary` path: create a Paperclip Project named `task-counter` on the test company, insert a `projectWorkspaces` row whose `cwd` points at the existing `cb7711f4-...` workspace (the one that already contains the Stage 5+6 commits). Assign all Stage 6.5 test issues under that project. Both Engineer and Designer then resolve to the same cwd via `project_primary`.

**Trade-off accepted:** the spec §2.5 "per-agent MCP isolation via `.mcp.json` in the agent's cwd" ambition is retired. True per-agent tool isolation does not exist in Paperclip today. Stage 6.5 isolates MCP usage via **skill assignment** instead: Engineer's `desiredSkills` does not include `ui-ux-pro-max`, so even if `mcp__21st-dev-magic__*` tools are present in Claude Code's registered tool list, Engineer's prompt never directs it to call them.

### 2.2 A9 direction → retract the `task-orchestration` needsDesignPolish-true branch; codify the explicit-Designer-slice pattern in `writing-plans`

The Stage 6 Tech Lead authored a plan with an explicit "Slice 4 — Designer Polish Subtask" instead of setting `needsDesignPolish: true` on slices 1-3. The explicit pattern is clearer (one-to-one slice→subtask mapping, no hidden flag routing), gives the TL subjective authority over "is this worth polishing?", and carries more information (scope, acceptance criteria, dependencies) than a boolean flag could.

Stage 6.5 retracts the `task-orchestration/SKILL.md § Stage 6 activation` amendment that spawned a Designer subtask from a `needsDesignPolish: true` flag, and amends `writing-plans/SKILL.md` to describe the explicit-slice pattern (prescriptive, not heuristic — the TL judges when to add a Designer slice). The `needsDesignPolish` flag is removed from the plan schema entirely; slices no longer carry it.

### 2.3 A10b (MCP startup) → pin version + pre-install + drop figma

Stage 6 evidence from Designer's heartbeat `8f442301` init event: `mcp_servers: [{"name":"@21st-dev/magic","status":"pending"}, {"name":"figma","status":"failed"}]`. The Magic server never transitioned from `pending` to `connected` within the heartbeat's execution window — likely because `npx -y @21st-dev/magic@latest` does a cold download per run and Claude Code proceeds to the model before startup completes. Figma failed immediately because `@figma/mcp` is not a real npm package.

Stage 6.5 fix:
1. Pin `@21st-dev/magic` to a concrete version (discovered at plan-execution time via `npm view`).
2. Pre-install the package globally in the `paperclip` container (`npm install -g @21st-dev/magic@<pinned>`) so `npx` resolves locally instead of round-tripping to the registry.
3. Rewrite `.mcp.json` to reference only `@21st-dev/magic`. Drop figma — wrong package name, unused for Stage 6.5 polish scope, can be revisited in Stage 7+ if Figma becomes relevant.
4. Place `.mcp.json` in the shared project workspace cwd (`cb7711f4-...`), not in Designer's agent_home (which is now bypassed).

**Trade-off accepted:** pre-installed packages survive container restarts but are lost on image rebuild. Stage 7's production rollout should bake the pinned version into the container image's Dockerfile. For Stage 6.5 (one validation run on a throwaway company), runtime install is enough.

No code movement needed. The path already has git history (f90947f → 15fc7fc), the task-counter project built out, and Stage 5+6 commits. The Stage 6.5 validation feature builds on top of HEAD.

## 3. §7.2 Rewrite (replaces the spec's current §7.2 verbatim)

> ### 7.2 Workspace model + MCP availability
>
> Paperclip resolves an agent's working directory via `resolveWorkspaceForRun` (server/src/services/heartbeat.ts). Priority order:
>
> 1. `project_primary`: issue has `projectId` AND a `projectWorkspaces` row with a valid `cwd` → all agents working on this project share that cwd.
> 2. `project_primary managed`: issue has `projectId` but no row → Paperclip auto-creates `/paperclip/instances/default/projects/<companyId>/<projectId>/<repoName>` (shared across agents on the project).
> 3. `task_session`: reuse the cwd stored in the previous session's params (only fires when the agent already ran on this issue).
> 4. `agent_home`: fallback → `/paperclip/instances/default/workspaces/<agentId>` (per-agent, isolated).
>
> The `claude_local` adapter's `adapterConfig.cwd` override is **only** honored when the resolved source is `agent_home` (claude-local adapter execute.ts `useConfiguredInsteadOfAgentHome`). When a project workspace is in play, `config.cwd` is silently ignored.
>
> **Paperclipowers uses the `project_primary` path.** For any feature that involves more than one role (Engineer + Designer, Engineer + QA, etc.), the issue chain MUST be created under a Paperclip Project with a `projectWorkspaces` row. All agents on that chain land in the shared cwd. This is the only mechanism Paperclip provides for cross-agent code sharing within a feature.
>
> ### MCP server configuration
>
> Place a `.mcp.json` file in the shared workspace cwd. Claude Code auto-loads it at session start and registers declared servers. `.mcp.json` format:
>
> ```json
> {
>   "mcpServers": {
>     "@21st-dev/magic": {
>       "command": "npx",
>       "args": ["-y", "@21st-dev/magic@<pinned-version>", "API_KEY=\"${MAGIC_API_KEY}\""]
>     }
>   }
> }
> ```
>
> **MCP servers are registered per-workspace, not per-agent.** Every agent whose heartbeat resolves to that cwd gets the same MCP tools in its Claude Code tool list. Per-agent MCP isolation at the runtime level does NOT exist in Paperclip as of this stage.
>
> **MCP startup discipline.** Claude Code's init event shows `mcp_servers: [{name, status}]` where status is one of `pending | connected | failed`. If a server is still `pending` when Claude Code proceeds to the first model turn, its tools never register. Guard against this with:
>
> - Pin package versions (avoid `@latest` — forces registry round-trip per cold start).
> - Pre-install packages globally in the container (`npm install -g <pkg>@<version>`) so `npx` resolves from the local install.
> - After each Stage 6.5+ validation run, verify the init event shows status `connected` for every MCP server the workflow depends on. Anything in `pending` or `failed` is a regression.
>
> ### Per-agent tool isolation without per-agent MCP
>
> Paperclipowers achieves the practical isolation goal (Engineer doesn't call Magic; Designer does) via **skill assignment**:
>
> - Engineer's `desiredSkills` includes `test-driven-development`, `systematic-debugging`, `verification-before-completion`, `code-review`, `pipeline-dispatcher`. None of these mention Magic or 21st.dev.
> - Designer's `desiredSkills` includes `ui-ux-pro-max` + `verification-before-completion` + `code-review` + `pipeline-dispatcher`. `ui-ux-pro-max` has explicit instructions to use `21st_magic_component_inspiration` and `21st_magic_component_refiner`.
> - Because each agent's system prompt is narrowed to its own skills, even when `mcp__21st-dev-magic__*` tools are registered, non-Designer agents have no prompt-level direction to invoke them.
>
> This is weaker than runtime tool-list scoping, but it is the only isolation Paperclip supports today, and for the Stage 6.5 feature set it is sufficient. Revisit if a future stage requires stronger guarantees (e.g., a Paperclip server PR to gate tool registration by `adapterConfig.allowedMcpTools`).
>
> ### Secrets
>
> 21st.dev Magic API key in Paperclip's secret store, name `21st-dev-magic`. Referenced via `envBindings` on the Designer agent: `{"MAGIC_API_KEY": {"type": "secret_ref", "secretId": "21st-dev-magic"}}`. The adapter resolves the secret at heartbeat time and exposes it to the MCP server subprocess via environment.

## 4. Skill Amendments

### 4.1 `task-orchestration/SKILL.md` — retract Stage 6 activation; add follower-reuse check (A6)

**Retract** (remove section § Stage 6 activation — spawning a Designer subtask when `needsDesignPolish: true`). Also remove § Reading the `needsDesignPolish` flag per slice.

**Replace** with a one-paragraph note in § The Process Step 2: "Each slice in the plan is one subtask. The Tech Lead authors the plan (including any explicit Designer polish slices, per `writing-plans/SKILL.md § Designer slices`); `task-orchestration` treats every slice uniformly — it does not inspect slices for polish flags or role heuristics. Whoever the plan says should own each slice is who gets assigned when that slice becomes the chain head."

**Add** follower-reuse pre-check to § Per-Completion Heartbeat Step 2 (A6 fix). New wording:

> Before POSTing any new subtask on a per-completion wake, query existing children: `GET /api/companies/<company-id>/issues?parentId=<parent-id>&limit=20`. A "pending child" has `status: "todo" | "in_progress"` AND is not yet terminal. For the slice you are about to spawn:
>
> 1. Determine the slice's expected predecessor subtask id(s) — the subtasks whose completion enables this one. Exactly the set you would set as `blockedByIssueIds` on a fresh POST.
> 2. Among pending children, find any whose `blockedByIssueIds` equals that set.
> 3. If EXACTLY ONE pending child matches: that child IS the next-in-line. PATCH its `assigneeAgentId` to the slice's intended assignee and keep its `blockedByIssueIds` as-is. DO NOT POST.
> 4. If ZERO pending children match: POST a fresh subtask as usual.
> 5. If TWO OR MORE match: abort. Post `@<board> ambiguous subtask graph for parent <id> — N candidates for the next slice, all blocked by <predecessor-ids>. Needs operator cancel-one-of cleanup.` and exit without POST or PATCH.
>
> The graph-based match is robust to title-format drift: it keys on the structural `blockedByIssueIds` + `parentId` shape rather than on any string convention. This protects against the "heartbeat rebuilds the subtask graph from scratch on wake" failure mode (Stage 6 Anomaly 6 — PAP-28 was created with `blockedByIssueIds: [PAP-27]`, then PAP-29 was POSTed with `blockedByIssueIds: []` and assignee pre-set, which under this new rule would trigger the "zero match" branch and still POST — except the explicit-slice amendment (§4.3) means the TL authors the Designer slice at plan time as a follower, so PAP-28-equivalent would have the right blockedBy and PAP-29 would be detected as a duplicate).

Update § Red Flags to include: "**Never** POST a new follower subtask from a per-completion wake without first querying existing children. If a pending follower already exists, PATCH it — POST creates the Stage 6 A6 duplicate-subtask bug."

### 4.2 `pipeline-dispatcher/SKILL.md` — Designer @mention requirement (A7)

Amend § If you are the Designer. Before the existing "Workflow inside the heartbeat" list, add a short **Notification Protocol** subsection:

> **Notification Protocol (required).** When your polish work reaches a terminal state, post a single comment on the subtask issue containing `@<tech-lead-name> DONE — <one-sentence summary>. Commits: <sha1>[, <sha2>...].` This @mention is the Tech Lead's wake signal for per-completion orchestration — without it, the pipeline deadlocks (Stage 6 Anomaly 7). On blocker, use `BLOCKED` instead of `DONE`; on ambiguity, `NEEDS_CONTEXT`. The Tech Lead's agent name is in the subtask description — do not @-mention yourself.

Cross-reference: `task-orchestration/SKILL.md § Notification Protocol — RULE 3` is producer-side (Tech Lead's instruction that goes INTO the subtask description). The pipeline-dispatcher amendment is consumer-side — it tells the Designer, who does not load `task-orchestration`, how to comply.

### 4.3 `writing-plans/SKILL.md` — codify explicit-Designer-slice pattern (A9)

Add a new subsection § Designer slices under whichever section describes slice authoring. Body:

> When a feature has UI output that would benefit from visual polish beyond what the Engineer's default styling produces, add an explicit Designer slice to the plan. Format:
>
> ```
> ### Slice N: Polish UI for <feature-name> (Designer)
>
> **Assignee role:** Designer
> **Dependencies:** Slice <engineer-slice-number(s)> (must be `done` before this slice starts)
>
> **Goal.** Visual polish of the UI produced by the Engineer slice(s). No backend or logic changes.
>
> **Acceptance criteria.**
> 1. All Engineer slice tests pass unchanged.
> 2. <visual criteria — specific to the feature, e.g., "uses the design system's token colors for table borders", "table is responsive below 640px">.
>
> **Required implementation files.** Same frontend files the Engineer modified (list explicit paths). Add new assets only if the polish requires them.
>
> **Notes.**
> - The Designer agent has `ui-ux-pro-max` in its skills and should use `21st_magic_component_inspiration` + `21st_magic_component_refiner` to ground decisions.
> - Do NOT use `21st_magic_component_builder` — it opens a browser and blocks in a headless container.
> ```
>
> The Tech Lead decides whether a Designer slice belongs in the plan. There is no heuristic and no plan-schema flag — if the feature's UI warrants polish, write the slice; if not, don't.

Remove any surviving reference to `needsDesignPolish` from the plan schema section. If that boolean is currently documented as a per-slice field, delete it.

### 4.4 `skills-paperclip/_shared/heartbeat-interaction.md` — unpause-before-PATCH rule (A11)

Add a new subsection § Unpause before PATCH. Body:

> When an operator action sequence involves both unpausing agents and PATCHing issue assignments that will fire a wake on those agents, the UNPAUSE must come FIRST. Paperclip's heartbeat scheduler silently drops wakes enqueued against paused agents (`server/src/services/heartbeat.ts` emits `WARN: failed to wake agent on issue update: Agent is not invokable in its current state` and does NOT re-fire the wake on subsequent unpause). Order:
>
> 1. For every agent that will receive a wake from the upcoming PATCHes: `POST /api/agents/<id>/resume` FIRST.
> 2. Verify `GET /api/agents/<id>` returns `status: "idle"` (not `paused`) for each.
> 3. THEN PATCH the issue assignments.
>
> If you PATCH-before-unpause, the pipeline stalls silently. Workaround if stalled: `PATCH /api/issues/<id>` with `{"assigneeAgentId": null}` then PATCH back to the intended assignee — the second PATCH re-fires the wake on the (now-unpaused) agent. Prefer the correct order.

Any Stage 7+ plan template that hires or unpauses agents must reference this file's § Unpause before PATCH.

### 5.1 Paperclip Project for `task-counter`

One-time setup on the Paperclipowers Test company. The project + its primary workspace can be created in a single call because `createProjectSchema` (`/app/packages/shared/src/validators/project.ts`) accepts an optional `workspace` sub-object:

   ```json
   {
     "name": "task-counter",
     "status": "active",
     "workspace": {
       "name": "task-counter-primary",
       "sourceType": "local_path",
       "cwd": "/paperclip/instances/default/workspaces/cb7711f4-c785-491d-a21a-186b07d445e7",
       "isPrimary": true
     }

   ```
   Capture the returned `project.id` as `$PROJECT_ID` and `project.workspaces[0].id` as `$PROJECT_WORKSPACE_ID`.
2. Verify: `GET /api/projects/$PROJECT_ID/workspaces` returns one row with `cwd` equal to the existing Engineer workspace path.
3. If the workspace needs to be created separately (e.g., because step 1 succeeded but workspace nesting failed): `POST /api/projects/$PROJECT_ID/workspaces` with the workspace sub-object alone.

For Stage 6.5 test issues, include `projectId: $PROJECT_ID` in the POST payload (createIssueSchema already accepts this field — verified in `/app/packages/shared/src/validators/issue.ts:40`). The server then uses `resolveWorkspaceForRun` priority 1 (`project_primary`) and returns `cwd = <that cwd>` for every agent woken on any issue in the chain, regardless of which agent is assigned.

## 7. Risks & Rollback

### 7.1 Risks

1. **Pre-installed Magic package is lost on image rebuild.** The `npm install -g` in the running container does not persist across `docker rm && docker create`. Stage 7 must bake the pinned version into the container image's Dockerfile. For Stage 6.5 (one-shot run), acceptable.
2. **`projectWorkspaces.cwd` pointing at an agent-UUID directory is semantically ugly.** If the Engineer agent is ever deleted and recreated, the workspace path goes stale. Mitigation: document this in the Stage 6.5 results doc as a Stage 7 action item ("move to a properly-named managed project workspace before production"). For Stage 6.5 throwaway company, acceptable.
3. **Non-Designer agents will now see Magic MCP tools in their Claude Code tool list.** Skill-level isolation (the agent's prompt doesn't mention them) is the only guard. If an agent's skill content ever inadvertently names Magic tools, that agent may invoke them. Stage 7 action item: linter check on skill content for cross-role tool references.
4. **Per-completion follower-reuse check (A6 fix) may match the wrong pending child** in edge cases where the plan has multiple Designer slices or multiple Engineer slices with similar titles. Mitigation: the check matches on slice identifier (if the subtask title includes "slice-N" prefix as the template prescribes), not just on role. If ambiguous, the TL prefers PATCH-an-existing-match over POST-a-new-one and escalates if no match is confident.

### 7.2 Rollback

If Stage 6.5 validation fails:

1. **Skill amendments** (§4) can be reverted via `git revert` on the commit that introduced them. The 8 paperclipowers skills in the library re-import from the reverted SHA (Stage 6's pin `e4ca9bb` stays valid as a fallback).
2. **Paperclip project** (§5.1) can be deleted via `DELETE /api/companies/<co>/projects/<id>` (or cascade via company deletion; the throwaway company is expendable).
3. **`.mcp.json`** in the shared workspace can be `rm`-ed; agents revert to pre-6.5 behavior.
4. **Pre-installed Magic package** can be `npm uninstall -g @21st-dev/magic` in the container; the package's only purpose is for this workflow.

None of these rollback steps affect other companies or other Paperclip features.
