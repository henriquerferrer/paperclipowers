### [NEW, A12] Reviewer rejection via status-only PATCH does not fire a wake

**Scenario.** Run 1 Phase K, Reviewer rejected PAP-31 by PATCHing `{status: "done" → "in_progress"}` while keeping `assigneeAgentId` unchanged (still pointing at Engineer). Paperclip's wake subsystem fired NO wake on Engineer — the Engineer sat idle for 6 minutes until operator nudge (unassign → reassign cycle) re-fired the `issue_assigned` wake.

**Evidence.** `agent_wakeup_requests` table showed 2 wake requests for Engineer in the PAP-31 window (both `issue_assigned`, both completed before rejection). No additional entry after Reviewer's rejection PATCH until the operator nudge created a new `199784d0` request.

**Root cause hypothesis.** Paperclip's wake fires on assignee-change events (`issue_assigned` → force-reset session) and @mention comments (`issue_comment_mentioned`). A status-only PATCH without assignee change doesn't produce either trigger. The Reviewer's pattern in `code-review § Per-subtask code review — if rejecting` says "PATCH `{status: in_progress, assigneeAgentId: <last-assignee>}`"; when the last assignee equals current, no `issue_assigned` wake fires.

**Stage 7 remediation candidates.** (a) `code-review` skill amends rejection to always null-then-reassign (two PATCHes with a sleep between — reliable but racier). (b) `code-review` skill uses the unassign → reassign idiom in ONE transaction if Paperclip supports it. (c) Server-side: fire `issue_status_changed` wake on any `todo/in_progress` transition even with same assignee. Option (c) is the cleanest but requires a Paperclip PR.

### [NEW, A13] Reviewer's `<board-name>` substitution is wrong

**Scenario.** Run 2 Phase C–D + E–G, both Reviewer approval comments end `@stage1-tester APPROVED — …`. `stage1-tester` is the Engineer's agent name, not the board. The PATCH's `assigneeUserId` routed correctly to the board user id (`Z1lWUQRX…`), but the @-mention in the comment body targeted the wrong agent.

**Evidence.** Two consecutive Reviewer comments in Run 2 used `@stage1-tester`. Run 1 had the same pattern (not explicitly grepped here but surfaced as part of the comment thread review).

**Root cause hypothesis.** `code-review/SKILL.md` instructs Reviewer to post `@<board-name> APPROVED`. The Reviewer's context has no reliable source for the board's @-name — the board is a Paperclip user with a better-auth user-id, not an agent with a clean display-name. The Reviewer's model picks an `@<name>` that looks "approver-ish" and happens to land on the first agent it thinks of — here, the Engineer.

**Stage 7 remediation.** (a) Configure the board user's display-name in a way that's discoverable from the Reviewer's skill context (e.g., `@board` resolves to the board user). (b) Amend `code-review` skill to NEVER @-mention the board — just post the findings comment + do the `assigneeUserId` PATCH. Board user is expected to check its queue; no explicit wake needed.
